## P0 — KRITISCH (Geldverlust / Systemausfall moeglich)

### [P0-1] Optimistic-Lock-Konflikt fuehrt zu Silent Fund Loss

**Datei:** `state.go` (Zeile ~454-476)

**Problem:** Bei `UPDATE WHERE version=$N` mit 0 affected rows (Konflikt) wird `acc.Version++` trotzdem erhoeht. Die In-Memory-Kopie denkt der Write war erfolgreich, obwohl er fehlschlug. Naechste Writes schlagen dauerhaft fehl — ohne Fehlermeldung nach oben.

**Angriffsszenario:** Zwei Nodes fuehren gleichzeitig Transfer aus. Node A gewinnt. Node B's Version-Counter laeuft aus dem Sync und verliert danach permanent die Faehigkeit, seinen State zu persistieren.

**Empfohlener Fix:**
```go
if rows == 0 {
    cs.db.QueryRow(`SELECT version FROM chain_accounts WHERE address = $1`, acc.Address).Scan(&acc.Version)
    return fmt.Errorf("optimistic lock conflict for %s, retry", acc.Address)
}
acc.Version++ // NUR nach erfolgreichem Write
```

### [P0-2] Pool-Distributions nicht atomar — Geldverlust bei Crash

**Datei:** `state.go` (DistributeValidatorsPool, DistributeLPPool, DistributeUBIPool)

**Problem:** Pool-Balance wird auf 0 gesetzt BEVOR alle Einzelverteilungen abgeschlossen sind. Crash nach Pool-Nullung = alle Rewards verschwinden, kein Rollback moeglich.

**Aktueller Code:**
```go
poolAcc.Balance = NewDecimal(0)
cs.saveAccountToDB(poolAcc)  // Pool geleert — KEIN ROLLBACK wenn naechste Zeilen schlagen fehl

for _, ns := range nodeShares {
    cs.saveAccountToDB(acc)  // Kann fehlschlagen!
}
```

**Fix:** Gesamte Verteilung in einer einzigen PostgreSQL-Transaktion.

### [P0-3] float64-Konvertierung beim DB-Write verliert Precision

**Datei:** `state.go` (Zeile ~450), `evm_storage.go` (Zeile ~487)

**Problem:** `Decimal` (int64 intern) wird via `.Float()` nach `float64` konvertiert, dann als `NUMERIC(20,6)` in Postgres geschrieben. float64 hat nur 15-16 signifikante Dezimalstellen — bei grossen Pools entstehen Rundungsfehler die sich akkumulieren.

**Fix:** Micro-Units (int64) direkt als `BIGINT` persistieren:
```sql
ALTER TABLE chain_accounts ADD COLUMN balance_micro BIGINT NOT NULL DEFAULT 0;
```

---

### [P0-4] AMMSwapOut Int64-Overflow bei Pools groesser als ~9000 AEQ

**Datei:** `decimal.go` (Zeile ~91-102)

**Problem:** Bei Reserves von ~9000 AEQ (= 9_000_000_000 Micro-Units) und grossem Swap:
`9_000_000_000 * 9_000_000_000 = 81 * 10^18` — ueberschreitet Int64-Max (~9.2 * 10^18).
Ergebnis wird negativ (wraparound) — Pool kann komplett geleert werden.

**Fix:** Bounds-Check nach big.Int-Berechnung:
```go
if result.BitLen() > 63 {
    return 0, fmt.Errorf("swap amount would overflow")
}
```

---

### [P0-5] Solidity: claimUBI erhoehrt totalSupply doppelt — Invarianten-Verletzung

**Datei:** `AequitasV7.sol` (Zeile ~264-275)

**Problem:** `accumulateUBI` zieht aus `ubiPool` ab. `claimUBI` addiert dann `owed` zu `balanceOf` UND zu `totalSupply`. Da das Geld schon in `totalSupply` (als Teil von `ubiPool`) war, wird es doppelt gezaehlt. Nach vielen Runden: `totalSupply >> SUM(balanceOf)`. `fairShare()` und `wealthCap()` werden progressiv falsch.

**Fix:** `totalSupply += owed` aus `claimUBI` entfernen — das Geld war schon in der Supply.

---

### [P0-6] Nonce-DB-Read unter Mutex blockiert alle anderen RPC-Handler

**Datei:** `evm_rpc.go` (Zeile ~385-412)

**Problem:** `s.state.LoadNonce()` (PostgreSQL-Query) wird unter `s.mu.Lock()` aufgerufen. Alle anderen RPC-Requests blockieren waehrend des DB-Reads. Bei hoher Last: De-facto Single-Threaded RPC-Server.

**Fix:** Nonce-Cache asynchron aufwaermen, DB-Read ausserhalb des kritischen Abschnitts.

---

### [P0-7] DB-Transaktionen mit READ COMMITTED statt SERIALIZABLE

**Datei:** `state.go` (beginStateTx)

**Problem:** `cs.db.Begin()` startet `READ COMMITTED` (PostgreSQL-Standard). Phantom-Reads moeglich. Invariante `totalSupply = humans * 1000` kann verletzt werden bei zwei parallelen Registrierungen.

**Fix:**
```go
tx, err := cs.db.BeginTx(ctx, &sql.TxOptions{Isolation: sql.LevelSerializable})
```

## P1 — HOCH (Sicherheitsluecken / Konsistenzprobleme)

### [P1-1] StateRoot() haelt RLock waehrend PostgreSQL-Query
**Datei:** `state.go`
`getConfigValue("last_ubi_at")` wird unter `cs.mu.RLock()` aufgerufen. DB-Latenz oder Deadlock blockiert alle State-Reads.

### [P1-2] PEER_SECRET als Klartext + Timing-Attack moeglich
**Datei:** `sync_blocks.go`
`subtle.ConstantTimeCompare` fehlt beim Secret-Vergleich. Timing-Analyse des String-Vergleichs kann das Secret offenbaren.

### [P1-3] Validator-Registrierung ohne Signatur-Verifikation
**Datei:** `api.go`
Wer `PEER_SECRET` kennt, kann beliebige `signing_address` als Validator eintragen. Kein Proof-of-Key-Possession = Konsens-Unterwanderung.

### [P1-4] Go-State und EVM-State divergieren bei nativen Transfers
**Datei:** `evm_rpc.go`
`SyncBalancesToEVM` liest Balance ohne Demurrage-Decay. EVM zeigt andere Werte als Go-State.

### [P1-5] Swap-Nonce Race Condition bei nonce=0
**Datei:** `evm_storage.go`
`INSERT ON CONFLICT DO NOTHING` fuer nonce=0 gibt keine verlaessliche Feedback-Information bei Race Conditions.

### [P1-6] last_ubi_at Update ausserhalb der Verteilungs-Transaktion
**Datei:** `state.go`
Crash zwischen letztem `saveAccountToDB` und `setConfigValue("last_ubi_at")` fuehrt zu doppelter UBI-Verteilung nach Restart.

### [P1-7] Balance in EVM-StateDB truncated auf Integer-AEQ
**Datei:** `evm_engine.go`
`int64(acc.Balance.Float())` schneidet alle Dezimalstellen ab. MetaMask zeigt falsche (abgerundete) Werte.

### [P1-8] Goroutine-Exhaustion durch unkontrollierte Peer-URL-Registrierung
**Datei:** `api.go`
Bekannte Validator-Adresse reicht um beliebige URLs zu registrieren und Sync-Goroutinen zu starten.

### [P1-9] StateRoot-Mismatch durch Wall-Clock-Abhaengigkeit
**Datei:** `block.go`
Demurrage-Timing-Unterschiede zwischen Nodes fuehren zu verschiedenen StateRoots. Gueltige Peer-Blocks werden dauerhaft abgelehnt.

### [P1-10] RegisterHuman + EVM-Sync nicht atomar
**Datei:** `state.go`, `register.go`
Crash zwischen Go-Write und EVM-Sync = dauerhaft inkonsistenter State.

### [P1-11] Potenzielle SQL-Injection in GetPriceHistory
**Datei:** `evm_storage.go`
```sql
-- Unsicher (auch wenn $1 int ist):
WHERE captured_at >= NOW() - ($1 || ' minutes')::INTERVAL
-- Sicher:
WHERE captured_at >= NOW() - ($1 * INTERVAL '1 minute')
```

### [P1-12] Solidity: triggerEscrowToUBI dekrementiert totalHumans inkonsistent
**Datei:** `AequitasV7.sol`
`fairShare()` berechnet sich falsch nach Escrow-Trigger da `totalHumans` sinkt, aber `totalSupply` nicht.

### [P1-13] loadFromDB ignoriert Scan-Fehler
**Datei:** `state.go`
`rows.Scan(...)` Rueckgabewert ungeprueeft. Schema-Mismatch nach Migration fuehrt zu Null-Wert-Accounts ohne Fehlermeldung.

### [P1-14] Swap-Nonce-Validierung in Swap-Handlern unklar
**Datei:** `api.go`
`ConsumeSwapNonce`-Infrastruktur existiert, aber ob sie in ALLEN Swap-Paths enforced wird, ist nicht eindeutig verifizierbar aus dem Code.

## Architektur-Hinweis

Das duale Ledger-Modell (Go-State + EVM-Storage) ist das fundamentale Design-Problem.
Beide Systeme repraesentieren zu verschiedenen Zeiten "die Wahrheit". Empfehlung:

1. Explizites Invarianten-Monitoring (Cronjob der beide Ledger vergleicht)
2. Alle Sync-Operationen in einen dedizierten Reconciliation-Service
3. Klare Read-Model vs Write-Model Dokumentation
4. Vollstaendige Integer-Unit-Migration (Micro-AEQ als BIGINT)
